# 灰度上线计划

## 目标

**安全地**把小模型从 0% 流量升到 100%，出问题能秒回滚。

## 前置条件（缺一不可）

- [ ] Benchmark 指标达上线阈值（见 `eval_spec.md` §5）
- [ ] 200 条黄金集人工评估均分 ≥ 4.0
- [ ] 压测通过：小模型 P95 ≤ 15s，Claude fallback P95 ≤ 30s
- [ ] Router + fallback 在测试环境跑通（主动拔网线测 fallback）
- [ ] Dashboard 上线且告警路径打通
- [ ] 业务方签字同意

## 灰度阶梯

| 阶段 | 流量占比 | 最短观察期 | 退出条件（进下阶段） |
|---|---|---|---|
| 0. Shadow | 0%（影子调用） | 3 天 | 影子输出 vs Claude 人工抽检分 ≥ 3.5/5 |
| 1. Canary | 5% | 3 天 | 关键指标不降，无 P0 事故 |
| 2. 小规模 | 20% | 4 天 | 同上 + 成本达预期 |
| 3. 中规模 | 50% | 7 天 | 同上 |
| 4. 大规模 | 80% | 5 天 | 同上 |
| 5. 全量 | 100% | 持续 | 持续监控 |

## 每个阶段必做

### 进入阶段前
1. 业务方 review 上一阶段 dashboard + 人工抽检
2. 填 `06_decisions/DP4_full_rollout.md` 对应子决策
3. 更新灰度比例配置

### 阶段运行中
- 每天早会 5 分钟看 dashboard
- 每天人工抽检 30 条
- 每次异常立即进 Runbook

### 进入下一阶段的硬指标

| 指标 | 阈值 |
|---|---|
| Fallback 率 | < 3% |
| 业务抽检采纳率 | > 85% |
| P95 时延 | < 15s（小模型） |
| 用户重试率 | 不高于历史 10% |
| 成本节省 | ≥ 40%（阶段 2+） |
| P0 事故 | 0 |

## 回滚机制

### 自动回滚触发条件
- Fallback 率 > 20%，持续 10 分钟
- vLLM 服务不可达，持续 2 分钟
- 业务报警（下游系统某指标劣化 > 30%）

### 手动回滚
- 业务方随时可以喊停
- 事故中值班工程师有权限直接降到 0%
- 回滚 = 把路由配置的 `small_pct` 改为 0，**不要删模型服务**

### 回滚后
1. 24 小时内出事故报告
2. 定位根因
3. 数据/模型回炉
4. 再次灰度从 5% 重新走

## 配置管理

灰度比例用配置中心（feature flag）管理，不写在代码里：
```yaml
# config/routing.yaml
small_model:
  enabled: true
  percentage: 5     # 改这里，秒级生效
  whitelist_task_types:
    - marketing_ai_v2_simple
    - marketing_ai_v2_mid
```

## A/B 测试设计（可选强化）

如果想量化小模型 vs Claude 的业务影响：
1. 随机分 50/50 两组
2. 跑 2 周
3. 对比：
   - 任务完成率
   - 用户重试率
   - NPS / CSAT（若有）
4. 显著性检验（chi-square 或 t-test）

**坑**：用户分组要按 user_id hash 稳定，不要每次随机——避免同一个用户来回切换体验。

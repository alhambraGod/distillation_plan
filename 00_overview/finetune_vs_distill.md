# 蒸馏 vs 微调 —— 一次讲清楚

> 客户最困惑的问题：**"我们到底是在做蒸馏还是微调？"**
> 一句话答案：**我们在做"用蒸馏思路的微调"**——两者不是对立的，而是组合使用。

---

## 1. 一张图看懂

```
                  把一个模型改得更适合你的业务
                              │
              ┌───────────────┴───────────────┐
              ▼                               ▼
          微调 (Fine-tuning)           蒸馏 (Distillation)
          回答"怎么改"                 回答"从哪学"
                                             │
          ┌───────┐                   ┌───────┴───────┐
          │ SFT   │                   │               │
          │ DPO   │                   ▼               ▼
          │ RL    │               教师监督            数据来源
          │ ...   │               (大→小)            (已有/生成)
          └───────┘
```

**结论**：
- 微调 = 改模型参数的**技术动作**（SFT / DPO / RL 等）
- 蒸馏 = 让小模型向大模型**学习的思想**
- 两者**正交**，经常叠加使用
- 本项目 = **"用合规教师知识 + SFT 技术"** = 蒸馏式微调。若教师知识来自 Claude 输出，必须先取得 Anthropic 书面许可或合同补充条款。

## 2. 严格定义

### 2.1 什么是微调（Fine-tuning）

**定义**：在预训练好的模型上，用新数据更新参数，让它在特定任务上表现更好。

**关键特征**：
- 起点：一个已经训好的**基座**（Qwen 7B / Llama 8B 等）
- 动作：在基座上继续训练（全参 / LoRA / QLoRA）
- 目标：改变模型的行为分布

**不关心"学谁"**：训练数据从哪来，是人工写的、别的模型生成的、还是你自己业务数据，微调都不管。

### 2.2 什么是蒸馏（Knowledge Distillation）

**定义**：用一个强模型（教师）的知识，训练一个弱模型（学生），让学生在特定任务上接近教师。

**关键特征**：
- 有两个模型：**教师**（大）+ **学生**（小）
- 学生的**训练信号**来自教师（不是来自人工标注）
- 目标：学生 ≤ 教师，但便宜 / 快

**不关心"用什么算法"**：可以用 SFT、KL 散度、DPO 等任一算法实现蒸馏。

### 2.3 两者关系

| 维度 | 微调 | 蒸馏 |
|---|---|---|
| 性质 | 技术动作 | 数据思想 |
| 必须有教师？ | ❌ 不需要 | ✅ 必须有 |
| 必须改参数？ | ✅ 必须 | ⚠️ 通常是（也有 prompt 蒸馏） |
| 数据来源 | 任意 | 必须来自教师 |
| 目标 | 改进模型在某任务上的表现 | 用小模型接近大模型 |

**包含关系**（文氏图）：

```
           所有改进模型的方法
┌─────────────────────────────────────────────┐
│                                             │
│   ┌──────微调 (Fine-tuning)──────┐          │
│   │                              │          │
│   │   ┌──────蒸馏的实现───────┐  │          │
│   │   │                       │  │          │
│   │   │  本项目                │  │          │
│   │   │  （SFT + 合规教师）   │  │          │
│   │   └───────────────────────┘  │          │
│   │                              │          │
│   └──────────────────────────────┘          │
│                                             │
│   Prompt-only 蒸馏（不改参数）              │
│                                             │
└─────────────────────────────────────────────┘
```

## 3. 本项目的准确定位

### 3.1 我们在做的是

```
本项目 = 黑盒输出蒸馏（Black-box SFT Distillation）

拆开看：
  ├── "微调"侧：用 SFT 算法 + LoRA 改学生参数
  └── "蒸馏"侧：训练数据来自教师模型、人工批准答案或已授权历史 trace
```

**为什么说是蒸馏而不只是微调**：
- 如果使用教师模型输出，数据不是人工从零标注，而是来自教师生成 / 审核
- 小模型的上限通常 = 教师和训练数据质量的上限
- 性质上是"学生向教师学"

**合规前提**：Claude 输出不能默认作为训练目标。只有 Anthropic 书面许可或合同明确允许时，才把 Claude trace 放进训练集；否则使用开源教师、人工批准答案或客户自有标准答案。

**为什么说也是微调**：
- 技术上就是 SFT（Supervised Fine-Tuning）
- 工具链是 `transformers + peft + trl`
- 产出是 LoRA adapter

### 3.2 如果不做蒸馏，只做微调

**场景**：你用**人工标注数据**（1000 条）训模型。
- 这是纯微调，不是蒸馏
- 数据质量取决于标注人
- 小模型上限不是 Claude，是标注质量

**场景**：你用**业务真实数据 + 正确答案对**训模型。
- 这是纯微调
- 需要自己准备"正确答案"

**问题**：以上两种都很贵 / 慢 / 数据少。**所以我们选蒸馏式微调**：让合规教师或业务专家替我们生成/验证答案。

### 3.3 如果只蒸馏不微调

**场景**：Prompt Distillation（提示蒸馏）
- 把 Claude 的多轮推理压缩成一句好 prompt 给学生
- 学生不训，只是推理时用这个 prompt
- 收益有上限，但**零训练成本**

**场景**：In-context Learning 蒸馏
- 把 Claude 的输出作为 few-shot 示例给学生
- 同样不训

**结论**：这类"不改参数"的蒸馏适合快速验证，但**无法长期替代微调**。

## 4. 两套技术栈对比

### 4.1 微调技术栈（不管教师）

| 层 | 技术 |
|---|---|
| 基座 | Qwen / Llama / Gemma / DeepSeek-base |
| 框架 | `transformers`, `peft`, `trl`, `unsloth` |
| 算法 | SFT / DPO / ORPO / KTO / RLHF / GRPO |
| 数据 | Alpaca 式 (instruction, input, output) / chatml / 偏好对 |
| 硬件 | 单卡 A100 / 消费级 3090 + QLoRA |
| 追踪 | W&B / TensorBoard |

### 4.2 蒸馏技术栈（需要教师）

| 层 | 技术 | 备注 |
|---|---|---|
| 教师 | 已授权 Claude API / Qwen 72B / DeepSeek V3 / Llama 70B | 参考 `teacher_model_comparison.md` |
| 蒸馏方式 | 黑盒 SFT / 白盒 KL / CoT 合成 / On-policy | 参考 `03_sft/distillation_techniques.md` |
| 学生 | Qwen 7B / Gemma 9B / Llama 8B | 参考 `base_model_selection.md` |
| 数据 pipeline | LangSmith / `langsmith` SDK / `distilabel` | 教师输出的收集/清洗 |
| 训练框架 | 和微调一样（transformers + peft + trl） | 本质还是微调 |
| 专用工具 | `DistillKit`（Arcee）, `SDG`（NVIDIA） | 合成数据 / 蒸馏工具包 |

### 4.3 关键差异（客户最常问）

| 问题 | 微调 | 蒸馏 |
|---|---|---|
| 数据哪里来？ | 自己准备 | 教师生成 |
| 成本主要在哪？ | 标注数据 | 教师推理 |
| 最大风险？ | 数据量不够 | 学生上限 ≤ 教师 |
| 适合谁？ | 有大量标注数据的团队 | 有强教师 API 或大模型权重的团队 |
| 我们是？ | — | ✅ 我们属于这类，但 Claude trace 只有获得书面许可后才能训练 |

## 5. 怎么使用（实操步骤）

### 5.1 纯微调流程（假设路线）

```
Step 1: 准备数据
  - 人工标 1000-10000 条 (instruction, output) 对
  - 或者业务数据打标

Step 2: 选基座
  - Qwen 7B / Llama 8B 等

Step 3: SFT 训练
  - LoRA / QLoRA
  - 1-3 epoch

Step 4: 评估 + 上线
```

**成本**：$20k-100k（标注）+ $2k-5k（算力）
**周期**：2-3 月（主要耗时在标注）

### 5.2 蒸馏流程（本项目）

```
Step 1: 选教师
  - 已授权 Claude trace，或 Qwen 2.5 32B/72B、DeepSeek 等开源/可商用教师
  - 参考 teacher_model_comparison.md

Step 2: 准备蒸馏数据
  - 从 LangSmith 拉已授权 trace，或用开源教师重新生成答案
  - 清洗 + 脱敏 + 分层（金/银/铜，见 data_quality_tiers.md）

Step 3: 选蒸馏方式
  - 黑盒 SFT / 白盒 KL / CoT
  - 参考 distillation_techniques.md

Step 4: 选学生
  - Qwen 7B / Gemma 9B 等

Step 5: 训练（和微调步骤一样）
  - LoRA + SFT 或 KL loss

Step 6: 评估 + 回退 + 上线
```

**成本**：$0-2k（数据，已有）+ $8k-15k（算力含教师推理）
**周期**：3 月
**优势**：标注成本几乎为 0

## 6. 技术难点与重点

### 6.1 微调的难点

| 难点 | 说明 | 解决 |
|---|---|---|
| 灾难性遗忘 | 学了业务任务，通用能力退化 | 混通用数据 20-30% |
| 过拟合 | 数据少，训过头 | 减 epoch / 加 dropout / early stop |
| 超参敏感 | lr 差一倍结果差 20% | 单变量实验 / W&B 追踪 |
| OOM | 显存不够 | QLoRA / grad checkpoint |
| 评估漂移 | 没有 benchmark 无法判断好坏 | 先做 eval_spec + 黄金集 |

### 6.2 蒸馏的额外难点（微调之外）

| 难点 | 说明 | 解决 |
|---|---|---|
| **幸存者偏差** | trace 只有"成功路径"，学生学到的天花板受限 | 数据分级（金/银/铜）+ 加难样本 |
| **教师上限继承** | 教师错的地方学生也错 | 人工黄金集做 sanity check |
| **Tokenizer 约束** | 白盒蒸馏要求同家族 | 选教师时同家族优先 |
| **License 合规** | 教师输出能否训第三方模型 | Legal review + 书面许可（见 `anthropic_tos_compliance.md`） |
| **数据质量分级** | 不是所有 trace 都能进训练集 | 金/银/铜分层（见 `data_quality_tiers.md`） |
| **分布对齐** | 训练分布 ≠ 线上分布 | On-policy DPO / 持续数据回流 |

### 6.3 本项目的特殊难点

| 难点 | 为什么这里特别难 |
|---|---|
| V1 / V2 架构不同 | trace schema 差异，必须物理隔离 |
| Agent 多轮 + 工具调用 | 现成评估框架不适用，必须自研 |
| Claude ToS | 未经 Anthropic 书面许可，用 Claude 输出训生成式/agent 小模型不应进入训练主线 |
| report_progress / submit_final_report 等业务原生协议 | 小模型必须学会这些"习惯" |
| 灰度回退 | Claude fallback 不是简单"挂了切过去"，要定义触发条件 |

## 7. 常见误区（给客户澄清）

### ❌ 误区 1："蒸馏就是微调的一种"
**真相**：蒸馏是数据思路，微调是技术动作。**蒸馏可以不微调**（prompt 蒸馏），微调也可以不蒸馏（纯人工数据）。

### ❌ 误区 2："小模型蒸馏出来能超越教师"
**真相**：**学生能力 ≤ 教师**。除非加 RL + 自我探索，否则不会超。

### ❌ 误区 3："有 trace 就能蒸馏"
**真相**：**脏数据和未授权数据都会让项目失败**。必须先数据分级 + 质量过滤 + license/ToS 审核。

### ❌ 误区 4："蒸馏不需要懂微调"
**真相**：蒸馏的执行 = 微调。你必须同时懂两套知识。

### ❌ 误区 5："微调只是 SFT"
**真相**：微调家族有 SFT / DPO / ORPO / KTO / RLHF / GRPO / DAPO 等多种。选哪个看问题。

### ❌ 误区 6："RL 是终极方案"
**真相**：RL 是**在前面都跑通后**的锦上添花。大多数场景 SFT + DPO 就够。详见 `rl_in_current_business.md`。

## 8. 本项目一句话定位

> 我们做的是：
> **用合规教师 / 人工批准答案产生的训练数据 + SFT 或 KL 蒸馏（微调技术）+ Qwen 7B（学生）= 可灰度上线的小模型**
>
> 在这个定位下：
> - **"蒸馏"** 回答了"数据从哪来"
> - **"微调"** 回答了"怎么训"
> - 两者缺一不可

1. **"蒸馏是'学谁'，微调是'怎么改'"** — 两者正交
